Função de Negócio: Design

As decisões de segurança mais impactantes e de menor custo são tomadas antes da escrita da primeira linha de código.

Nível 0: Sem Planejamento ("Sai codando")

Descrição Aprimorada: A equipe de desenvolvimento recebe os requisitos funcionais e inicia a implementação imediatamente. Não há uma pausa para considerar os riscos de segurança inerentes à nova funcionalidade. A arquitetura é puramente orientada pela funcionalidade, e as vulnerabilidades de design são inevitavelmente introduzidas, tornando a correção futura muito mais cara e complexa.

Atividades Típicas:

  • Codificar funcionalidades diretamente a partir de requisitos de negócio.
  • Não realizar análises de risco ou modelagem de ameaças.
  • Problemas de segurança no design só são descobertos após a implementação, geralmente por meio de um incidente.

Benefícios:

Nenhum benefício de segurança. A velocidade inicial é rapidamente anulada pelo alto custo de retrabalho para corrigir falhas de arquitetura.

Critérios de Autoavaliação (Checklist):

  • ☐Nossa equipe começa a codificar assim que os requisitos funcionais são definidos?
  • ☐Não temos um processo para discutir os potenciais riscos de segurança de uma nova feature?

Recomendações para o Próximo Passo: Introduza sessões curtas e informais de "whiteboard hacking" ou modelagem de ameaças ad-hoc para novas funcionalidades críticas. Use o mnemônico STRIDE para fazer perguntas simples como "Como alguém poderia falsificar sua identidade aqui?" ou "Quais informações sensíveis estamos expondo?".

Nível 1: Modelagem de Ameaças Ad-Hoc ("E se...?")

Descrição Aprimorada: A equipe começa a fazer perguntas sobre segurança proativamente durante a fase de design. Em sessões curtas e focadas, geralmente em um quadro branco, os desenvolvedores debatem os potenciais abusos de uma nova funcionalidade. O objetivo não é ter todas as respostas, mas sim levantar as questões certas e identificar os riscos mais óbvios antes do desenvolvimento.

Atividades Típicas:

  • Realizar sessões de 30 minutos de "whiteboard hacking" para features críticas.
  • Usar o mnemônico STRIDE (Spoofing, Tampering, etc.) para guiar a discussão de riscos.
  • Documentar as ameaças identificadas e as possíveis mitigações nos tickets de desenvolvimento.

Benefícios:

  • Prevenção: Identifica falhas de design antes que se tornem código, economizando tempo e dinheiro.
  • Conscientização: Ensina a equipe a pensar como um atacante.
  • Baixo Custo: Requer apenas tempo da equipe, sem a necessidade de ferramentas especializadas.

Critérios de Autoavaliação (Checklist):

  • ☐Para novas funcionalidades importantes, paramos para discutir "o que pode dar errado" em termos de segurança?
  • ☐Já usamos um framework simples como o STRIDE para guiar essa discussão?
  • ☐As ameaças identificadas são documentadas de alguma forma?

Recomendações para o Próximo Passo: Formalize essa prática adicionando "Requisitos de Segurança" explícitos como critérios de aceite nos tickets de desenvolvimento. Por exemplo, para uma história sobre login, um critério seria "A API deve bloquear um usuário após 5 tentativas de login falhas".

Nível 2: Requisitos de Segurança ("Precisamos garantir que...")

Descrição Aprimorada: As discussões sobre ameaças são traduzidas em requisitos concretos e verificáveis. A segurança se torna um critério de aceite explícito nas histórias de usuário, assim como os requisitos funcionais. Isso garante que a segurança seja tratada como uma parte essencial da funcionalidade, e não como um extra opcional.

Atividades Típicas:

  • Adicionar uma seção de "Critérios de Aceite de Segurança" nos templates de tickets do Jira, Trello, etc.
  • Escrever requisitos claros, como "A API deve validar que o usuário autenticado só pode visualizar seus próprios dados".
  • Validar esses requisitos durante a revisão de código e os testes de QA.

Benefícios:

  • Clareza: Os desenvolvedores sabem exatamente quais controles de segurança precisam implementar.
  • Rastreabilidade: Cria um vínculo claro entre o risco identificado e a solução implementada.
  • Qualidade: Garante que a "feature" só é considerada "pronta" quando está funcional e segura.

Critérios de Autoavaliação (Checklist):

  • ☐Nossos tickets de desenvolvimento incluem critérios de aceite específicos para segurança?
  • ☐Verificamos o cumprimento desses critérios antes de considerar uma tarefa concluída?
  • ☐Os requisitos de segurança são específicos e testáveis?

Recomendações para o Próximo Passo: Vá além de requisitos por ticket e construa uma "arquitetura segura por padrão". Desenvolva templates de projetos, bibliotecas internas ou "scaffolds" que já incorporem os controles de segurança mais comuns, tornando o caminho seguro o caminho mais fácil para o desenvolvedor.

Nível 3: Arquitetura Segura por Padrão ("O jeito fácil é o jeito seguro")

Descrição Aprimorada: Neste nível, a segurança é incorporada na própria fundação da arquitetura de software. A equipe investe na criação de componentes, bibliotecas e templates reutilizáveis que já vêm com controles de segurança configurados por padrão. Isso reduz a carga cognitiva sobre os desenvolvedores, pois eles não precisam se lembrar de implementar controles de segurança básicos a cada novo serviço ou funcionalidade; o jeito certo já é o jeito padrão.

Atividades Típicas:

  • Criar um template para novos microsserviços que já inclua logging estruturado, headers de segurança e tratamento de exceções.
  • Desenvolver uma biblioteca interna para lidar com autenticação e autorização de forma centralizada.
  • Manter um "catálogo de padrões de segurança" com exemplos de código seguros para tarefas comuns.

Benefícios:

  • Escalabilidade: A segurança escala junto com o desenvolvimento, sem se tornar um gargalo.
  • Consistência: Garante que os controles de segurança sejam aplicados de forma uniforme em toda a aplicação.
  • Eficiência: Libera os desenvolvedores para focar na lógica de negócio, sabendo que a base é segura.

Critérios de Autoavaliação (Checklist):

  • ☐Temos templates de projeto ou microsserviços que incluem configurações de segurança por padrão?
  • ☐Usamos bibliotecas internas para centralizar lógicas críticas de segurança (ex: autorização)?
  • ☐O desenvolvedor precisa fazer um esforço extra para não ser seguro?

Recomendações para o Próximo Passo: Com uma arquitetura segura por padrão, o foco se volta para a validação contínua e a pesquisa de ameaças emergentes. A equipe pode explorar a criação de "padrões de abuso" específicos para o seu negócio e integrá-los aos processos de design.
