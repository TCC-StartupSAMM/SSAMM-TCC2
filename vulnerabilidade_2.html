Vulnerabilidade #2: A Cadeia de Suprimentos de Software (Dependências Vulneráveis)

O Risco

"Seu aplicativo é composto por 70% a 90% de código de terceiros. Você confia em cada um dos seus fornecedores de código?".

Cada npm install ou pip install é um ato de confiança que importa não apenas a funcionalidade desejada, mas também todas as vulnerabilidades contidas naquela biblioteca e em suas dependências transitivas.

Cenário de Ataque (Log4Shell - CVE-2021-44228)

Uma vulnerabilidade de execução remota de código foi descoberta em uma biblioteca de logging Java onipresente, a Log4j. Milhões de aplicações, de servidores de Minecraft a sistemas corporativos complexos, tornaram-se instantaneamente vulneráveis. As equipes de segurança entraram em uma corrida desesperada não para corrigir o próprio código, mas para simplesmente descobrir se e onde estavam usando a biblioteca vulnerável.

Defesas Pragmáticas

Ação: A defesa mais eficaz e de menor esforço contra essa ameaça é a implementação de SCA (Software Composition Analysis). Ferramentas de SCA escaneiam os arquivos de manifesto do projeto (como package-lock.json ou requirements.txt), identificam todas as dependências e as cruzam com bancos de dados de vulnerabilidades conhecidas (CVEs).

Ferramentas:

A barreira de entrada para SCA é praticamente zero.

  • Dependabot do GitHub: É gratuito, nativo do GitHub e ativado por padrão na maioria dos repositórios. Ele não apenas alerta sobre dependências vulneráveis, mas também abre Pull Requests automaticamente com a correção, exigindo apenas que um desenvolvedor revise e aprove o merge.
  • Snyk: Oferece um plano gratuito robusto que se integra diretamente ao repositório Git e ao pipeline de CI/CD, fornecendo análises detalhadas e sugestões de correção.
